# News Data Collector

Проект представляет собой гибкую, модульную систему для сбора, обработки и представления новостных данных (или аналогичной информации) из различных источников. Реализован в рамках учебного курса по объектно-ориентированному программированию, демонстрируя применение принципов SOLID, паттернов проектирования, конкурентных и асинхронных моделей, веб-парсинга, создания Telegram-бота и тестирования.

Проект прошёл **13 этапов разработки**, каждый из которых добавлял новые концепции и улучшал архитектуру. Ниже представлен итоговый результат.

## Установка и настройка

1. **Убедитесь, что установлен Python 3.11+** и [uv](https://docs.astral.sh/uv/#installation).
2. **Клонируйте репозиторий** и перейдите в папку проекта:
   ```bash
   git clone <url>
   cd project_<фамилия>_<группа>
Установите зависимости:

bash
make install
# или uv sync
Для работы Telegram-бота создайте файл .env на основе .env.example и укажите токен:

text
TELEGRAM_BOT_TOKEN=ваш_токен
Токен можно получить у @BotFather.

Использование
Консольный запуск
Запуск сбора данных и вывод результата в консоль:

bash
make project
# или uv run python -m src.main
Конфигурация (источники, стратегия, режим) задаётся в src/main.py через словарь, передаваемый в AppRunner.

Telegram-бот
Установите токен в переменную окружения и запустите:

bash
# Linux/macOS
export TELEGRAM_BOT_TOKEN="ваш_токен"
# Windows PowerShell
$env:TELEGRAM_BOT_TOKEN="ваш_токен"
uv run python -m src.bot
Бот поддерживает команды:

/start – приветствие и список команд.

/collect – запустить сбор и обработку данных (использует текущую конфигурацию).

/sources – выбрать источники (демо, файл, веб).

/strategy – выбрать стратегию (нормализация, фильтрация).

/mode – выбрать режим выполнения (sequential, threads, processes, async, hybrid).

/stats – показать результат последнего запуска.

Выбор режима выполнения
В конфигурации (main.py или через бота) можно задать:

sequential – последовательное выполнение.

threads – пул потоков (подходит для I/O-bound задач).

processes – пул процессов (для CPU-bound задач).

async – чистый асинхронный режим (asyncio).

hybrid – асинхронный сбор + executor для тяжёлой обработки.

Архитектура
Основные компоненты
Модель данных (NewsItem) – единица информации с метаданными, перегруженными операторами сравнения (__str__, __repr__, __lt__, __eq__, __hash__).

Источники данных – реализуют общий интерфейс (синхронный или асинхронный) и поставляют поток объектов.

Стратегии обработки – инкапсулируют алгоритмы (нормализация, фильтрация, тяжёлые вычисления).

Презентер – отвечает за форматирование вывода (консоль).

Приложение (HybridApp) – координирует сбор, обработку и вывод, может использовать executor.

Раннер (AppRunner) – единая точка входа, загружает конфигурацию и создаёт компоненты.

TaskManager – централизованное управление пулами потоков/процессов.

Паттерны проектирования
Singleton – AppConfig для глобальной конфигурации.

Factory – SourceFactory создаёт источники по типу.

Strategy – различные реализации ProcessingStrategy.

Abstract Base Classes – формализованы интерфейсы источников и стратегий.

Iterator / Generator – источники возвращают генераторы, обработка потоковая.

Context Manager – файловые источники используют with для гарантированного закрытия.

Режимы выполнения
Последовательный – простейший, для отладки.

Потоки (threads) – параллельная обработка нескольких источников (I/O).

Процессы (processes) – параллельные вычисления на нескольких ядрах.

Асинхронный (asyncio) – неблокирующие операции, эффективен при множестве ожиданий.

Гибридный – асинхронный ввод-вывод + executor для тяжёлой синхронной работы.

Веб-парсинг
В проекте реализован асинхронный веб-источник (AsyncWebNewsSource), который:

Выполняет конкурентные запросы к API (пример – JSONPlaceholder).

Ограничивает параллелизм семафором.

Повторяет запросы при временных ошибках (retries).

Соблюдает задержки между запросами (политика вежливости).

Передаёт данные потоково, без сохранения в файл (в отличие от части 7, где использовался отдельный парсер).

Тестирование
Тесты написаны с использованием pytest и покрывают:

Модели (перегрузка операторов, создание).

Стратегии (нормализация, фильтрация).

Источники (демо, файловый, веб) с моками.

Парсер (с подменой HTTP-ответов).

Гибридное приложение.

Запуск тестов:

bash
make test
# или uv run pytest
Производительность
При тестировании на 30 записях из веб-источника были получены следующие результаты (пример):

Последовательный режим: 0.35 с

Потоки: 0.31 с

Процессы: 0.50 с

Асинхронный: 0.30 с

Гибридный: 0.28 с

Потоки и асинхронность дают выигрыш при наличии задержек (I/O), процессы – при интенсивных вычислениях. Для небольших объёмов данных разница незначительна.

Этапы разработки
Часть 1. Введение в ООП – базовая структура, классы NewsItem, MockNewsSource, NewsProcessor, App.

Часть 2. Наследование и композиция – базовый класс BaseDataSource, два конкретных источника, композиция в App.

Часть 3. Принципы SOLID – выделен ConsolePresenter, обработчик стал иммутабельным, рефакторинг.

Часть 4. Абстрактные классы и перегрузка операторов – формальные интерфейсы, перегрузка __lt__, __eq__, __hash__.

Часть 5. Паттерны (Factory, Strategy, Singleton) – AppConfig, SourceFactory, ProcessingStrategy, две стратегии.

Часть 6. Итераторы, генераторы, with – потоковая обработка, файловый источник читает построчно, контекстные менеджеры.

Часть 7. Веб-парсинг – WebNewsParser сохраняет данные в файл, файловый источник их читает.

Часть 8. Конкурентность (потоки/процессы) – добавлены режимы sequential, threads, processes, замеры времени.

Часть 9. Асинхронность (asyncio) – асинхронные источники, AsyncNormalizationStrategy, асинхронное приложение.

Часть 10. Гибридная модель – TaskManager, HybridApp, вынос тяжёлых вычислений в executor.

Часть 11. Углублённый веб-парсинг – асинхронный веб-источник с контролем нагрузки, повторными попытками, передачей данных напрямую.

Часть 12. Telegram-бот – внешний интерфейс, использующий AppRunner, команды управления.

Часть 13. Тестирование – модульные тесты для всех ключевых компонентов.
